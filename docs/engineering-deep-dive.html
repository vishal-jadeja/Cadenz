Cadenz — architecture, decisions
& tradeoffs

A personal branding and productivity platform: an AI layer over a user's intellectual life that happens to make posting effortless. This document walks through the stack, the schema, the security model, and — more importantly — the specific tradeoffs made and why, including the ones that didn't fully resolve.

01

Product & Philosophy

What Cadenz is, deliberately, and what it refuses to be.

Cadenz (renamed from an earlier working title, "Mintmark") takes its name from the stamp a mint presses into a coin to certify authenticity and origin. Tagline: "Stamp your knowledge on the internet." The product observes what a user is actually doing — commits, session logs, notes — and surfaces it as personal-brand content, rather than asking the user to invent something to post.

Core philosophy
"Observe first. Understand always. Act only when ready."

Restated as the design constraint behind every feature: the app must deliver value to a user who never manually posts anything. Passive observation is the product; publishing is an optional output on top of it.

Explicit anti-positioning, stated directly in the project's own planning docs:

  • Not a social media scheduler (Buffer/Hootsuite category)
  • Not a content-generation machine
  • Not a general-purpose AI chatbot
  • Not a Pomodoro / time-tracking tool
  • Not an analytics dashboard — post performance is never a user-facing feature

Two build phases, one already shipped:

Phase 1 — Early Access ✅ complete

Waitlist landing page, referral-weighted queue position, invite-only account creation, admin dashboard for managing the invite pipeline.

Phase 2 — Main App 🟡 in progress

Onboarding, unified activity heatmap, Notes, Settings, Content Studio, AI Assistant — with Notion sync, a browser extension, and public portfolios queued for later phases.

Tech Stack & Rationale

Every choice below was made for a specific reason stated in the project's own build docs — not defaults picked out of habit.

LayerChoiceWhy
Framework Next.js 16.2, React 19.2, TypeScript (strict) Next 16 renamed middleware.ts → proxy.ts and changed enough conventions that the project's own AGENTS.md flags it as "not the Next.js you know" — a reminder to verify against installed docs rather than training data.
Server state TanStack Query v5 Single source of truth for anything that comes from an API. Hard rule: never useState + useEffect to fetch data.
UI / cross-component state Zustand v5 Patches a specific gap in TanStack Query: mutations aren't shared across hook instances. If component A triggers a mutation, component B has no built-in way to know it succeeded. Zustand flags (e.g. uiStore.waitlistJoined) are the signal.
HTTP client Shared Axios instance (withCredentials: true) One instance, everywhere — except the AI assistant's streaming hook, which uses native fetch() because Axios doesn't support streaming response bodies. Documented exception, not an inconsistency.
Database Supabase (Postgres + Row Level Security) RLS enabled on every table with no exceptions, treated as a non-negotiable baseline rather than a per-table judgment call.
Auth NextAuth v5 (beta), Credentials provider, JWT sessions Invite-only signup means there's no external OAuth identity for the core account — JWT avoids a database session table for a login path that's just email + password.
Email Brevo + React Email Switched from Resend mid-build. Kept as a visible decision in the docs rather than silently overwritten — the early spec still references Resend, later steps reference Brevo.
Rate limiting / cache Upstash Redis Per-endpoint sliding/fixed-window limits (waitlist join, invite accept, OAuth connect, BYOK key save all have distinct limits) plus TTL caching for heatmap/admin stats reads.
Background jobs Trigger.dev v3 Chosen explicitly over hand-rolled queues: gives parallelism, triggerAndWait, retries, timeouts, and observability "and nothing else is needed."
AI orchestration Native TypeScript + Trigger.dev tasks, BYOK adapter (Anthropic/OpenAI/Gemini/Groq) Explicit rejection of agent frameworks (no LangGraph, no CrewAI, no LangChain) — the orchestration need is a bounded fan-out/critic/revise loop, not a general agent runtime. BYOK also means Cadenz never pays for a user's AI usage.
Heatmap / graph rendering D3.js Explicit rule against pulling in a third-party heatmap library for what is a 52×7 grid and a force-directed topic graph — both cheap to own directly.
Deployment Vercel + Supabase + Trigger.dev Cloud API function timeout raised 10s → 30s in vercel.json specifically for batch invite sends and OAuth token exchange, which don't fit the default window.

Query defaults

Set once in QueryProvider.tsx and deliberately conservative:

staleTime: 5 * 60 * 1000   // 5 minutes
retry: 1
refetchOnWindowFocus: false
refetchOnReconnect: false
// no global refetchInterval — polling is opt-in per hook

Polling only exists where there's a real async job to watch: GitHub backfill status (5 retries, 3s apart) and Content Studio generation status (2s interval, switched off once the job reaches a terminal state).

State Management Split

Three tiers, one rule each — not a matter of taste, a matter of where the data actually lives.

Server state
API data, mutations
→
TanStack Query
src/lib/queries/
Cross-component UI state
flags, filters, wizard step
→
Zustand
src/stores/
Local ephemeral state
inputs, toggles, one-time reads
→
useState only

The Zustand layer exists to solve one narrow, concrete problem rather than a general "global state is convenient" instinct: TanStack Query mutations are not shared across hook instances. If the waitlist form triggers useJoinWaitlist(), the success page component rendered elsewhere in the tree has no way to know that happened — hence uiStore.waitlistJoined set inside the mutation's onSuccess and read wherever else it's needed.

Three stores exist today: uiStore (waitlist-joined flag, command palette, sidebar), adminStore (admin table filters + pagination), onboardingStore (wizard step, connected platforms, BYOK-skipped flag).

Database & Row Level Security

Every table has RLS. Every mutation is also filtered by user_id in application code first — RLS is the backstop, not the only guard.

Core tables (Phase 1)

TablePurpose
usersCore account row. password_hash populated only via the accept_invite_account RPC; role gates admin access.
waitlistSignup queue: email, name, auto-generated referral_code, referred_by, status (waiting / invited / joined).
invite_tokensSingle-use, 48-hour-expiry tokens tying an email to an invite.
user_settingsOne row per user (PK = FK). Theme, timezone, active_platforms, onboarding progress. Admins never get a row here — the proxy redirects them away from any page that assumes one exists.
system_configKey/value runtime config (invite cap, referral bonus). Service-role read/write only.

The referral bonus function

The standout piece of SQL in the project — the entire "refer 5 friends, jump 5 spots per referral, floor at position 1" business rule lives in a single Postgres function, mirrored in application code (REFERRAL_SLOTS_BONUS = 5 in src/lib/config.ts) with a comment that the two must be kept in sync:

CREATE OR REPLACE FUNCTION public.get_waitlist_position(p_email text)
RETURNS integer LANGUAGE plpgsql STABLE AS $$
DECLARE
  v_referral_code  text;
  v_base_position  integer;
  v_referral_count integer;
  v_effective      integer;
BEGIN
  SELECT referral_code, row_number() OVER (ORDER BY created_at ASC)::integer
  INTO v_referral_code, v_base_position
  FROM public.waitlist WHERE email = p_email;

  IF NOT FOUND THEN RETURN NULL; END IF;

  SELECT COUNT(*)::integer INTO v_referral_count
  FROM public.waitlist WHERE referred_by = v_referral_code;

  -- Each referral moves the person 5 spots forward, minimum position 1
  v_effective := GREATEST(1, v_base_position - (v_referral_count * 5));
  RETURN v_effective;
END;
$$;

Atomic invite acceptance

Account creation from an invite token has to satisfy two constraints at once: never let two concurrent requests redeem the same token, and never burn the token if account creation fails partway through. Both are handled inside a single SECURITY DEFINER function using row locking, rather than as separate API-route steps:

CREATE OR REPLACE FUNCTION public.accept_invite_account(
  p_token text, p_name text, p_password_hash text, p_timezone text DEFAULT 'UTC'
) RETURNS json LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  v_invite  public.invite_tokens%ROWTYPE;
  v_user_id uuid;
BEGIN
  -- 1. Lock the token row — prevents concurrent double-redemption
  SELECT * INTO v_invite FROM public.invite_tokens
  WHERE token = p_token AND used_at IS NULL AND expires_at > now()
  FOR UPDATE;

  IF NOT FOUND THEN
    RETURN json_build_object('error', 'invalid_token');
  END IF;

  IF EXISTS (SELECT 1 FROM public.users WHERE email = v_invite.email) THEN
    RETURN json_build_object('error', 'already_registered');
  END IF;

  INSERT INTO public.users (email, name, password_hash)
  VALUES (v_invite.email, p_name, p_password_hash)
  RETURNING id INTO v_user_id;

  INSERT INTO public.user_settings (user_id, timezone, active_platforms)
  VALUES (v_user_id, p_timezone, '[]');

  -- token marked used_at + waitlist.status = 'joined' follow in the same transaction
  ...
END;
$$;

This replaced an earlier two-step API-route design (verify token → create user as separate requests) once the concurrent-redemption edge case was identified — the RPC makes the whole operation one transaction instead of two round trips with a race condition between them.

Avoiding N+1 in the admin list

get_admin_waitlist computes a per-row referral count with a self-join and GROUP BY in the same query that fetches the page, rather than one count query per row:

       COUNT(w2.id)::integer AS referral_count
FROM public.waitlist w
LEFT JOIN public.waitlist w2 ON w2.referred_by = w.referral_code
WHERE (p_status IS NULL OR w.status = p_status)
  AND (p_search IS NULL OR w.email ILIKE '%' || p_search || '%')

Design note

Platform connections (observation — GitHub, Gmail) and active platforms (publishing — LinkedIn, X, Medium) are treated as two separate concepts everywhere: separate tables, separate UI sections, never conflated in an API response. Connecting GitHub says nothing about whether a user wants to publish there — because they can't; GitHub isn't a publishing target at all.

Phase 12 — Notes

folders and notes, with notes.folder_id set to ON DELETE SET NULL — deleting a folder unfiles its notes rather than deleting user content. embedding vector(1536), notion_page_id, and notion_synced_at are intentionally absent from this schema, not just unused — pgvector search and Notion sync are Phase 2 additions that didn't need speculative columns sitting empty until then.

Auth & Security

JWT sessions, a proxy that gates by role before any page code runs, and envelope encryption for anything sensitive at rest.

Session flow

Credentials provider
→
bcrypt.compare()
via service-role client
→
JWT
id + role claim
→
proxy.ts
role-based routing

NextAuth's jwt() callback stashes id and role on the token at sign-in; session() copies both onto session.user. That's what lets the proxy make a routing decision from the JWT alone, with no per-request database round trip.

Proxy-level route protection

Next.js 16 renamed middleware.ts to proxy.ts. It wraps every request in auth((request) => {...}) and enforces three rules before any page component runs:

  • /admin and /api/admin/*: no session → redirect/401; authenticated but non-admin → redirect/403 (deliberately not a 404, to avoid confirming the route exists via an inconsistent error, but also to avoid leaking existence through 404-vs-403 timing) — the actual doc rationale given is to prevent route enumeration.
  • General app routes (/dashboard, /onboarding, /notes, /api/user, /api/connections, /api/notes, /api/folders, /api/activity): unauthenticated → redirect to /login?callbackUrl=....
  • An authenticated admin hitting a user page (/dashboard, /onboarding, /notes) gets redirected to /admin — this is the actual mechanism behind "admins have no user_settings row": they're never allowed to reach a page that assumes one exists.

Key handling

KeyRoleUsed in
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEYAnon key, respects RLS, safe in the browserclient.ts, server.ts
SUPABASE_SECRET_KEYService role, bypasses RLS admin.ts only, via plain createClient() — not the SSR helper, since it's a stateless server-only client with no concept of a user session

The SDK derives both the apikey and Authorization: Bearer headers from the key passed to createClient() — a documented rule against overriding those manually in global.headers, since doing so would silently break auth in a way that's easy to miss in review.

Encryption at rest

OAuth tokens (platform_connections) and BYOK keys (api_keys) are encrypted with AES-256-GCM using Node's built-in crypto — no extra dependency for something the standard library already does well. Format is iv:authTag:ciphertext, base64-encoded. The API-key GET route is written to never return the decrypted value, "not even to the authenticated user" — the UI can confirm a key exists and show its provider, never its contents.

Other concrete measures

  • Honeypot field on the waitlist form positioned off-screen with left: -9999px rather than display: none, because bots specifically check for the latter.
  • Per-endpoint rate limits sized to the endpoint's risk: waitlist join 10/hr/IP, invite accept 5/hr, BYOK key save 20/min (flagged sensitive), Content Studio generation 10/min (expensive — runs against the user's own key).
  • PKCE specifically for LinkedIn and X OAuth; GitHub/Gmail/Medium use standard OAuth2.
  • Gmail's OAuth scope is locked to gmail.readonly at the scope-request level, not just enforced in application code afterward.
  • Account deletion deletes across tables in an explicit, documented order — largest/most dependent tables first, users row last — rather than relying on a single cascading FK chain that can hold locks too long on large tables.
06

Phase 1 — Early Access System

Live and complete: waitlist, referral queue, invite acceptance, admin dashboard.

Waitlist & referral tracking shipped

Landing page embeds the waitlist form and a live count (/api/waitlist/count, cached 60s via Cache-Control). Visiting /ref/[code] sets a 30-day, non-httpOnly referral_code cookie (non-httpOnly because the form needs to read it client-side); the form silently attaches it as referred_by on submit — invisible to the referred user.

Success-state copy branches on position: users already inside the top REFERRAL_SLOTS_BONUS (5) spots see "you're near the top" instead of a "move up faster" CTA that would make no sense for them.

Invite acceptance shipped

/invite/[token] verifies the token server-side before rendering, then passes it to the client form as a prop rather than a hidden input — kept out of the DOM where it isn't necessary. Token consumption is atomic with account creation via accept_invite_account (see Database section) so a failed signup never burns a valid invite.

Admin dashboard shipped

Five API routes: stats, paginated waitlist, single invite, batch invite, config. Batch invites clamp the requested count to remaining capacity under getEarlyAccessLimit() rather than either erroring outright or silently over-sending past the cap. Batches larger than 20 are handed to a Trigger.dev job instead of processed inline, so the HTTP response doesn't time out — small batches stay synchronous, large ones go async.

Implementation drift, logged not hidden

The original spec assumed a simple bearer-token (ADMIN_SECRET) gate for /admin. The shipped version replaced this with full NextAuth JWT role-based protection instead — a stronger mechanism, and the deviation is called out explicitly in the build docs rather than left as an undocumented change.

07

Phase 2 — Main App

Onboarding through to the AI assistant — mostly shipped, two features still at the spec stage.

Onboarding wizard shipped

8 sub-phases: DB schema extension, wizard shell, OAuth connections (5 platforms), GitHub commit backfill, active-platform selection with per-platform AI instructions, first manual session log, optional BYOK step, and the dashboard scaffold itself.

GitHub backfill (90 days of commit history via Trigger.dev, paginated Search API calls, upserted into unified_activity) is treated as critical path rather than a nice-to-have — it's what makes the dashboard non-empty on day one instead of showing a blank heatmap to a brand-new user.

The BYOK step is optional by design and built to feel non-punitive: skip is never hidden, never de-emphasized, never gated behind a second confirmation. A missing key later surfaces as a gentle prompt, not an error.

Dashboard scaffold: D3-rendered 52×7 heatmap, streak counter as a pure function (no side effects — current streak counts consecutive days back from today, longest streak is a sliding window), and topic distribution shown as a ranked bar list rather than a pie chart — a deliberate legibility choice over a chart type that's harder to compare precisely.

Background jobs shipped

send-batch-invites exists alongside the original synchronous admin route rather than replacing it — the sync route stays fine for the admin dashboard's interactive flow, the Trigger.dev version exists for anything that needs to trigger sends without a human watching. cleanup-expired-tokens runs 2am UTC daily and specifically excludes users who hold a different, still-valid token when reverting expired invites back to waiting — a correctness detail that avoids clobbering a legitimately reissued invite. daily-intelligence and topic-extraction are registered stubs that return {status: 'not_implemented'} so Phase 2 work can fill in the body without touching config or registration.

Settings page shipped

Pure UI work — every backing API route already existed from onboarding. Connections tab flattens the onboarding's two-section (observe/publish) layout into one flat list of five platforms, since Settings doesn't need the onboarding narrative. Publishing tab uses an initialized guard so a background refetch can't clobber an in-progress edit — a concrete fix for a React Query + local form state race. API keys tab uses a two-step delete with a 3-second auto-revert window instead of a modal.

Notes page shipped

Split-pane editor; NoteEditor is keyed by note id specifically to force a remount on switch and avoid stale local state carrying over between notes. isDirty is computed by diffing local state against the fetched note rather than pulling in a form library for what's effectively one textarea and a title field. Markdown preview dynamically imports marked to keep it out of the SSR bundle, and renders via dangerouslySetInnerHTML — accepted as safe because it's a user's own content rendered back to only that user, not sanitized against cross-user XSS because there's no cross-user rendering path here.

Content Studio spec'd, not built

Introduces the shared BYOK adapter (createByokAdapter(userId)) used by every AI feature from this point on — returns null rather than throwing when no key is configured, pushing call sites toward graceful degradation instead of a try/catch wall. Provider fallback order is fixed: Anthropic → OpenAI → Gemini → Groq.

Planned pipeline: a research agent pulls notes + recent posts, fans out per active platform only, a critic agent scores each draft against platform format rules, and a reviser agent runs on failure — capped at 2 revision loops, then ships what it has. A bounded-cost decision over chasing a perfect draft indefinitely.

AI Assistant spec'd, not built

Reuses the Content Studio's BYOK adapter rather than a second AI abstraction. Retrieval for v1 is plain ILIKE text search over notes (plus "always include the 5 most recent notes regardless of match"), explicitly staged as a stepping stone to real pgvector cosine similarity search later — both queries are documented side by side in the spec as "now" and "Phase 2." Every retrieval query is scoped by user_id in application code before RLS ever gets a chance to apply. Missing-key state returns HTTP 402 with a structured error rather than a generic 500, and the UI renders a full "add your key" screen instead of an error state. Streaming is a hand-rolled 4-event SSE protocol, decoded manually since Axios can't stream. Context is hard-capped — 12 notes, 10 topics, 30 days of activity — stated directly as a cost-control measure, not a technical ceiling.

Key Tradeoffs

Decisions with a stated cost and a stated reason for accepting it.

Bounded revision over perfect output

Content generation caps critic/reviser loops at 2 passes, then ships whatever exists. Chasing a perfect draft indefinitely would make generation latency and cost unbounded for a feature that runs on the user's own API key — a worse failure mode than an occasionally imperfect post.

Fan-out scoped to active platforms

Generation only ever targets platforms the user has explicitly marked active, never all five connected platforms. Fewer platforms means fewer AI calls, which means lower cost against a user-supplied key — directly stated as the reasoning, not an incidental side effect.

One cron, fan-out inside it

The daily intelligence job is a single scheduled task that iterates all users, not one scheduled task per user. Avoids the operational cost of registering/monitoring N schedules as the user base grows.

Debounced AI work on notes

Topic extraction waits 10 minutes after the last note save; knowledge-graph recompute waits 5. Triggering either on every keystroke or every save would multiply AI calls for no benefit, since a note is rarely "final" the instant it's saved.

Text search now, vectors later

The assistant ships with ILIKE retrieval instead of waiting for pgvector to be wired in. Explicitly staged as an interim step with the upgrade path already written down, rather than blocking the whole feature on embeddings infrastructure.

Sync route kept alongside the async one

The admin dashboard's synchronous invite-send route wasn't replaced when the Trigger.dev version shipped — both exist because they serve different call patterns (interactive vs. batch/automated), and removing the sync path would have added a network hop to a request that doesn't need one.

No third-party heatmap or agent framework

D3 directly for the heatmap and topic graph; native TypeScript + Trigger.dev tasks instead of LangGraph/CrewAI for the AI pipeline. Both are cases where the "framework" would add an abstraction layer over something narrow enough to own outright — a 52×7 grid and a fan-out/critic/revise loop, not an open-ended agent runtime.

Known Limitations & Future Work

What's explicitly out of scope today, and what's deferred on purpose.

Not implemented yet

Content Studio and AI Assistant are fully spec'd but unbuilt. Daily intelligence and topic extraction are stub tasks returning not_implemented. pgvector embeddings columns don't exist yet — added only when the feature needing them ships.

Manual infra steps

Production Vercel/Supabase/Upstash/Trigger.dev-cloud provisioning, OAuth app credentials, and Brevo domain verification are code-complete but not yet executed against production — tracked explicitly as a checklist, not silently assumed done.

Deliberately excluded, permanently

No analytics dashboard — post performance isn't a user-facing feature even once the data exists. No live Pomodoro/timer widget — Cadenz tracks activity, it doesn't manage time.

Explicitly deferred to later phases

Notion two-way sync (conflict resolution: most-recent updated_at wins, flagged if within 5 minutes), a browser extension with strict "raw URLs never leave the browser" privacy rules, public portfolio pages, third-party integrations (LeetCode, Readwise).

Retrospective — what I'd flag in review

Two places where the build docs and the shipped code disagree with each other. Worth naming directly rather than pretending the paper trail is perfectly clean.

Doc drift — proxy cookie refresh

CLAUDE.md specifies that the proxy should call supabase.auth.getUser() on every request to refresh SSR session cookies, and never use the unverified getSession(). The shipped src/proxy.ts only performs the NextAuth JWT check — the Supabase cookie-refresh call isn't there. Either it needs to be added, or the SSR client's own getAll/setAll cookie handling in server.ts already covers it and the instruction is stale. This is exactly the kind of gap I'd want a second engineer to catch in review rather than assume away.

Scope inconsistency — Pomodoro

The README's feature list still includes a Pomodoro timer under a planned "Productivity Suite," while the project's own scope guard states plainly: "Cadenz tracks activity, it does not manage time... a live timer widget is out of scope." One of the two needs to change — I'd rather surface that tension than let a reviewer find it first.

Neither of these is a shipped bug — they're both documentation-vs-implementation gaps caught by re-reading the project's own specs against its own code, which is the point of writing this document in the first place.
